# Phase 5.1: Monitoring & Operations

## Overview

This document defines the operational framework for the agentic tokenomics system in production. The operational philosophy follows a layered approach: autonomous agents handle routine decisions (Layer 1-2), structured monitoring detects anomalies before they become incidents, and human operators intervene only when escalation thresholds are breached or governance-level decisions are required.

The system is designed around four operational principles:

1. **Observability-first** -- every agent decision, smart contract state change, and economic metric is instrumented before the feature ships
2. **Automated alerting** -- known failure modes have automated detection with progressive severity; novel patterns surface via anomaly detection
3. **Minimal manual intervention** -- agents handle routine decisions, runbooks codify operator actions, and auto-remediation covers common failures
4. **Progressive response escalation** -- issues start at the lowest-cost resolution path and escalate only when automated or operator-level responses are insufficient

### PromQL Alert Rules

```yaml
# alerting_rules.yml
groups:
  - name: agent_health
    interval: 30s
    rules:
      # --- Agent Down (>5 minutes) ---
      - alert: AgentDown
        expr: regen_agent_up == 0
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "Agent {{ $labels.agent_id }} ({{ $labels.agent_name }}) is down for >5 minutes"
          runbook: "See Runbook: Agent Restart/Recovery (RUN-001)"

      # --- High Error Rate ---
      - alert: AgentHighErrorRate
        expr: >
          rate(regen_agent_error_total[5m])
          / rate(regen_agent_decisions_total[5m]) > 0.05
        for: 5m
        labels:
          severity: P2
        annotations:
          summary: "Agent {{ $labels.agent_id }} error rate above 5%"
          runbook: "Check agent logs; verify MCP connectivity"

      # --- Escalation Rate >20% ---
      - alert: AgentHighEscalationRate
        expr: >
          rate(regen_agent_escalation_total[1h])
          / rate(regen_agent_decisions_total[1h]) > 0.20
        for: 15m
        labels:
          severity: P2
        annotations:
          summary: "Agent {{ $labels.agent_id }} escalation rate above 20%"
          runbook: "Review escalation reasons; check data quality; see Playbooks"

      # --- Low Confidence Trend ---
      - alert: AgentLowConfidenceTrend
        expr: >
          histogram_quantile(0.5,
            rate(regen_agent_decision_confidence_bucket[1h])
          ) < 0.6
        for: 30m
        labels:
          severity: P3
        annotations:
          summary: "Agent {{ $labels.agent_id }} median confidence below 0.6"

      # --- SLA Response Time Breach ---
      - alert: AgentSLAResponseTimeBreach
        expr: >
          histogram_quantile(0.95,
            rate(regen_agent_workflow_duration_seconds_bucket[1h])
          ) > 86400
        for: 5m
        labels:
          severity: P2
        annotations:
          summary: "Agent {{ $labels.agent_id }} workflow {{ $labels.workflow_id }} p95 latency exceeds SLA"

  - name: economic_health
    interval: 60s
    rules:
      # --- Supply Cap Violation (P0) ---
      - alert: SupplyCapViolation
        expr: regen_supply_current > regen_supply_hard_cap
        for: 0s
        labels:
          severity: P0
        annotations:
          summary: "CRITICAL: Circulating supply exceeds hard cap"
          runbook: "Immediate halt of mint operations; engage core team (P0-001)"

      # --- Validator Set Below Minimum (P0) ---
      - alert: ValidatorSetBelowMinimum
        expr: sum(regen_authority_validators_active) < regen_validator_min_required
        for: 1m
        labels:
          severity: P0
        annotations:
          summary: "CRITICAL: Active validators ({{ $value }}) below minimum (15)"
          runbook: "Emergency governance escalation; contact offline validators (P0-002)"

      # --- Fee Conservation Failure (P0) ---
      - alert: FeeConservationFailure
        expr: >
          abs(
            rate(regen_fee_collected_total[5m])
            - sum(rate(regen_fee_distributed_total[5m])) by ()
          ) > 1
        for: 5m
        labels:
          severity: P0
        annotations:
          summary: "CRITICAL: Fee collection != fee distribution; conservation violated"
          runbook: "Halt fee router; audit M013 state (P0-003)"

      # --- Supply Mint/Burn Ratio Drift ---
      - alert: SupplyMintBurnDrift
        expr: >
          abs(regen_supply_minted_period - regen_supply_burned_period)
          / (regen_supply_minted_period + regen_supply_burned_period + 1) > 0.5
        for: 168h  # 1 week
        labels:
          severity: P2
        annotations:
          summary: "M012 mint/burn ratio significantly imbalanced for 7+ days"

      # --- Low Fee Revenue ---
      - alert: LowFeeRevenue
        expr: rate(regen_fee_collected_total[24h]) == 0
        for: 48h
        labels:
          severity: P2
        annotations:
```

### P0: Critical Incidents

#### P0-001: Supply Cap Violation (M012)

```yaml
incident_id: P0-001
trigger: SupplyCapViolation alert (regen_supply_current > regen_supply_hard_cap)
impact: "Economic invariant broken; trust in token monetary policy compromised"

immediate_response:
  time_target: <5 minutes
  steps:
    1: "On-call operator acknowledges alert in PagerDuty"
    2: "Execute emergency halt of x/supply mint operations"
    3: "Verify alert is not a metrics reporting error (query ledger directly)"
    4: "If confirmed: freeze all token transfers via emergency governance proposal"
    5: "Page core team (all hands)"

investigation:
  steps:
    1: "Identify the transaction(s) that caused supply to exceed cap"
    2: "Check for parameter manipulation (was hard_cap changed?)"
    3: "Audit recent governance proposals for unauthorized parameter changes"
    4: "Review M012 algorithm execution logs for the period"
    5: "Check for integer overflow or rounding errors in mint calculation"

remediation:
  steps:
    1: "If excess is from incorrect mint: execute corrective burn"
    2: "If parameter was manipulated: revert via governance"
    3: "Deploy code fix if algorithm error identified"
    4: "Resume operations only after independent verification"

post_incident:
  - "Publish post-mortem within 48h"
  - "Update security invariant tests"
  - "Review M012 Security Invariant 1 (Cap Inviolability) enforcement"

communication:
  - "Public status page: immediate update"
  - "Discord #operations + #validators: real-time updates every 15m"
  - "Validator notification: direct contact via Signal/phone"
  - "Governance forum: post-mortem within 48h"

escalation_path: |
  On-call → Core Engineering → Foundation Board
  Notify: All validators, community Discord, governance forum
```

### Runbook: Agent Restart/Recovery

```yaml
runbook_id: RUN-001
name: Agent Restart/Recovery
estimated_duration: 5-15 minutes
prerequisites:
  - kubectl access to production cluster
  - Agent pod names known (format: regen-agent-{001-004}-{replica})

procedure:
  step_1:
    action: "Identify the failed agent pod"
    command: |
      kubectl get pods -n regen-agents -l agent_id=$AGENT_ID
      kubectl describe pod $POD_NAME -n regen-agents

  step_2:
    action: "Check recent logs for error cause"
    command: |
      kubectl logs $POD_NAME -n regen-agents --tail=200
      kubectl logs $POD_NAME -n regen-agents --previous --tail=200

  step_3:
    action: "Determine restart strategy"
    decision_tree: |
      If OOMKilled:
        → Increase memory limits (step_4a)
      If CrashLoopBackOff:
        → Check config and dependencies (step_4b)
      If ImagePullBackOff:
        → Check image registry (step_4c)
      If healthy but unresponsive:
        → Delete pod for fresh restart (step_4d)

  step_4a:
    action: "Increase memory limits"
    command: |
      kubectl patch deployment regen-agent-$AGENT_ID -n regen-agents \
        -p '{"spec":{"template":{"spec":{"containers":[{"name":"agent","resources":{"limits":{"memory":"4Gi"}}}]}}}}'

  step_4b:
    action: "Check configuration and dependencies"
    command: |
```

### Runbook: Agent Key Rotation

```yaml
runbook_id: RUN-004
name: Agent Key Rotation
estimated_duration: 30-60 minutes
prerequisites:
  - New keypairs generated offline
  - Governance proposal for on-chain key update (if applicable)
  - Scheduled maintenance window

procedure:
  step_1:
    action: "Generate new ed25519 keypair for agent"
    command: |
      # Generate offline, never on production server
      openssl genpkey -algorithm Ed25519 -out agent_$AGENT_ID_new.pem
      openssl pkey -in agent_$AGENT_ID_new.pem -pubout -out agent_$AGENT_ID_new.pub

  step_2:
    action: "Register new public key on-chain (if agent has on-chain identity)"
    command: |
      regen tx authz grant $NEW_AGENT_ADDRESS \
        generic --msg-type=$AUTHORIZED_MSG_TYPE \
        --from=$ADMIN_KEY \
        --expiration=$(date -d "+1 year" +%Y-%m-%dT%H:%M:%SZ) \
        --chain-id=regen-1 \
        -y

  step_3:
    action: "Update Kubernetes secret"
    command: |
      kubectl create secret generic regen-agent-$AGENT_ID-keys \
        --from-file=private.pem=agent_$AGENT_ID_new.pem \
        --from-file=public.pem=agent_$AGENT_ID_new.pub \
        --dry-run=client -o yaml | kubectl apply -f - -n regen-agents

  step_4:
    action: "Rolling restart of agent pods"
    command: |
      kubectl rollout restart deployment regen-agent-$AGENT_ID -n regen-agents
      kubectl rollout status deployment regen-agent-$AGENT_ID -n regen-agents

  step_5:
    action: "Verify agent authenticates with new key"
    command: |
      kubectl logs -l agent_id=$AGENT_ID -n regen-agents --tail=20 | grep -i "auth"

  step_6:
    action: "Revoke old key authorization"
    command: |
      regen tx authz revoke $OLD_AGENT_ADDRESS $AUTHORIZED_MSG_TYPE \
        --from=$ADMIN_KEY \
        --chain-id=regen-1 \
        -y

  step_7:
    action: "Securely destroy old private key"
    command: |
      shred -vfz -n 5 agent_$AGENT_ID_old.pem
```
