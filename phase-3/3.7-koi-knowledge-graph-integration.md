# Phase 3.7: KOI Knowledge Graph Integration

## Overview

KOI (Knowledge Organization Infrastructure) is Regen Network's knowledge graph backed by Apache Jena (RDF/SPARQL) and Qdrant (vector embeddings). It serves as the semantic memory layer for all 4 agent personas, providing entity resolution, document search, and audit trail storage. KOI currently contains 64,004 documents and 78,354 code entities across 15 entity types.

This specification defines how the agentic tokenomics system integrates with KOI at every layer: ontology design for Regen-specific domain concepts, data flow pipelines for ingesting on-chain and off-chain data, SPARQL query patterns for each agent's OODA loops, MCP tool mappings to concrete workflows, audit object schemas with provenance guarantees, and infrastructure configuration for Apache Jena and Qdrant.

## Data Flow Architecture

### Pipeline Overview

```
+---------------------------------------------------------------------------+
|                          KOI DATA FLOW                                     |
+------------------+----------------------+----------------------------------+
|  ON-CHAIN -> KOI |   AGENT -> KOI       |   EXTERNAL -> KOI               |
|  (per-block)     |   (immediate)         |   (daily)                       |
|                  |                       |                                  |
|  Tendermint WS   |   OODA Act Phase      |   Discourse API                 |
|      |           |       |               |   GitHub API                    |
|      v           |       v               |   Methodology PDFs              |
|  Event Indexer   |   Decision Object     |       |                         |
|      |           |       |               |       v                         |
|      v           |       v               |   Crawler + Parser              |
|  RDF Transformer |   KOI Object Builder  |       |                         |
|      |           |       |               |       v                         |
|      v           |       v               |   Embedding Pipeline            |
|  SPARQL Update   |   KOI MCP create      |       |                         |
|      |           |       |               |       v                         |
|      v           |       v               |   Qdrant + Jena                 |
|  Apache Jena     |   Apache Jena         |                                 |
|  + Qdrant embed  |   + Qdrant embed      |                                 |
+------------------+----------------------+----------------------------------+
```

### Pipeline 1: On-Chain to KOI

```yaml
pipeline_id: PIPE-ONCHAIN
name: On-Chain Event Indexer
trigger: Tendermint WebSocket (new block events)
latency: < 5 seconds per block
throughput: ~1 block/6.5s (Regen Ledger block time)

source:
  type: tendermint_websocket
  endpoint: "ws://localhost:26657/websocket"
  subscribe_queries:
    - "tm.event='NewBlock'"
    - "tm.event='Tx'"

event_types_indexed:
  ecocredit:
    - MsgCreateClass -> regen:CreditClass
    - MsgCreateProject -> regen:Project
    - MsgCreateBatch -> regen:CreditBatch
    - MsgRetire -> update regen:retiredAmount
    - MsgUpdateClassAdmin -> update regen:classAdmin
    - MsgUpdateProjectAdmin -> update regen:projectAdmin
    - MsgProposeClassCreator -> regen:CreditClass (pending)

  governance:
    - MsgSubmitProposal -> regen:Proposal
    - MsgVote -> regen:Vote
    - proposal_status_change -> update regen:proposalStatus
    - EndBlocker (proposal finalized) -> update regen:tallyResult

  staking:
    - MsgCreateValidator -> regen:Validator
    - MsgEditValidator -> update regen:moniker, regen:commission
    - slash_event -> update regen:uptimeScore
    - validator_set_update -> update regen:votingPower

  attestation_bond:  # M008
    - MsgCreateAttestation -> regen:Attestation
    - MsgChallengeAttestation -> update regen:bondStatus
    - MsgResolveChallenge -> update regen:bondStatus

  service_escrow:  # M009
    - MsgCreateAgreement -> regen:ServiceAgreement
    - MsgCompleteMilestone -> update regen:completedMilestones
    - MsgResolveDispute -> update regen:agreementStatus

  reputation:  # M010
    - MsgSubmitSignal -> regen:ReputationSignal
    - MsgRevokeSignal -> delete regen:ReputationSignal

transform:
  processor: rdf_transformer
  steps:
    - parse_event_attributes
    - map_to_ontology_class
    - generate_iri (deterministic from chain_id + tx_hash + msg_index)
    - add_provenance (block_height, tx_hash, timestamp)
    - serialize_turtle

load:
  target: apache_jena
  method: SPARQL_UPDATE
  endpoint: "http://localhost:3030/regen/update"
  graph: "<https://regen.network/graph/onchain>"
  batch_size: 50  # triples per SPARQL UPDATE
  retry:
    max_attempts: 3
    backoff: exponential
    initial_delay: 1s

embedding:
  trigger: after_successful_load
  model: BAAI/bge-large-en-v1.5
  collection: regen_onchain
  fields_to_embed:
    - rdfs:label
    - dcterms:description
    - regen:proposalType (for proposals)
```

### Immutability Guarantees

```yaml
immutability_policy:
  storage:
    - KOI objects are content-addressed (IRI derived from content hash)
    - Once stored, the content cannot be modified
    - Updates create new objects with prov:wasRevisionOf links

  verification:
    - content_hash is computed before storage and verified on read
    - provenance.data_sources are resolvable IRIs
    - agent_character_hash can be verified against the deployed character config

  audit_chain:
    - Each decision links to its trigger event (on-chain tx or schedule)
    - Evidence IRIs form a verifiable dependency graph
    - Human overrides create a new audit object with override metadata
    - The full chain is queryable via SPARQL (see Q-AUDIT queries below)

  retention:
    - Audit objects are never deleted
    - Storage cost is managed via Jena TDB2 compaction
    - Cold storage migration after 1 year (Jena -> archival graph)
```

### Hybrid Search Pattern

The hybrid search pattern combines SPARQL structural queries with Qdrant semantic similarity to produce ranked results that satisfy both graph constraints and meaning-based relevance.

```
+-----------------------------------------------------------------------+
|                      HYBRID SEARCH PIPELINE                            |
|                                                                        |
|  User Query / Agent Context                                            |
|       |                                                                |
|       +----------------------+---------------------------+             |
|       v                      v                           |             |
|  +--------------+   +------------------+                 |             |
|  | SPARQL Filter |   | Vector Embedding |                 |             |
|  | (structural)  |   | (semantic)       |                 |             |
|  |              |   |                  |                 |             |
|  | Graph pattern |   | BGE-large-en    |                 |             |
|  | matching      |   | 1024-dim encode |                 |             |
|  +------+-------+   +--------+---------+                 |             |
|         |                    |                           |             |
|         v                    v                           |             |
|  candidate_iris[]     similarity_scores[]                |             |
|         |                    |                           |             |
|         +--------+-----------+                           |             |
|                  v                                       |             |
|         +----------------+                               |             |
|         | Score Fusion   |  score = a * sparql_rank      |             |
|         | (RRF or linear)|        + b * cosine_sim       |             |
|         +--------+-------+  a=0.4, b=0.6 (tunable)      |             |
|                  v                                       |             |
|         Ranked Results                                   |             |
+-----------------------------------------------------------------------+
```

#### Implementation

```typescript
interface HybridSearchParams {
  query: string;                    // natural language query
  sparqlFilter?: string;            // optional SPARQL WHERE clause for structural constraints
  collection: string;               // Qdrant collection name
  limit: number;                    // max results
  alpha?: number;                   // SPARQL rank weight (default 0.4)
  beta?: number;                    // vector similarity weight (default 0.6)
  filterPayload?: Record<string, unknown>;  // Qdrant payload filter
}

async function hybridSearch(params: HybridSearchParams): Promise<SearchResult[]> {
  const { query, sparqlFilter, collection, limit, alpha = 0.4, beta = 0.6 } = params;

  // Step 1: SPARQL structural filter (if provided)
  let candidateIRIs: string[] | null = null;
  if (sparqlFilter) {
    const sparqlResults = await koiMCP.sparqlQuery(`
      SELECT ?entity WHERE { ${sparqlFilter} }
    `, limit * 3);  // over-fetch for re-ranking
    candidateIRIs = sparqlResults.map((r: any) => r.entity);
  }

  // Step 2: Qdrant vector search
  const vectorResults = await qdrantClient.search(collection, {
    vector: await embedText(query),
    limit: limit * 3,
    filter: candidateIRIs ? {
      must: [{ key: "entity_iri", match: { any: candidateIRIs } }]
    } : params.filterPayload ? params.filterPayload : undefined,
    with_payload: true,
  });

  // Step 3: Reciprocal Rank Fusion
  const sparqlRanks = new Map<string, number>();
  if (candidateIRIs) {
    candidateIRIs.forEach((iri, idx) => sparqlRanks.set(iri, 1 / (60 + idx)));
  }

  const fusedScores = vectorResults.map((result: any) => {
    const iri = result.payload.entity_iri;
    const vectorScore = result.score;
    const sparqlScore = sparqlRanks.get(iri) || 0;
    return {
      iri,
      score: alpha * sparqlScore + beta * vectorScore,
      payload: result.payload,
    };
  });

  fusedScores.sort((a: any, b: any) => b.score - a.score);
  return fusedScores.slice(0, limit);
}
```

```yaml
      ttl: 300s  # 5 minutes
      max_memory: 128MB

    vector_search_cache:
      description: "Cache hybrid search results by query embedding hash"
      key_format: "koi:vector:{sha256(embedding)}"
      ttl: 60s   # short TTL due to dynamic scoring

  invalidation:
    on_graph_update:
      - flush keys matching "koi:sparql:*" for the updated graph
      - flush entity caches for modified entities
    on_ontology_change:
      - flush all caches (full invalidation)

      - Redis SCAN + DEL with pattern matching
```

### Pagination Patterns

```yaml
pagination:
  # SPARQL OFFSET/LIMIT pattern for standard result sets
  standard:
    max_page_size: 100
    default_page_size: 25
    pattern: |
      SELECT ?s ?p ?o
      WHERE { ... }
      ORDER BY ?s
      LIMIT ${page_size}
      OFFSET ${(page - 1) * page_size}

  # Cursor-based pagination for large or frequently updated result sets
  cursor_based:
    description: "Use last-seen entity IRI as cursor for stable pagination"
    pattern: |
      SELECT ?entity ?property ?value
      WHERE {
        ?entity a regen:AgentDecision ;
                dcterms:created ?created .
        FILTER (?created < "${cursor_timestamp}"^^xsd:dateTime)
      }
      ORDER BY DESC(?created)
      LIMIT ${page_size}
    advantages:
      - Stable under concurrent inserts (no skipped/duplicate rows)
      - Efficient for time-ordered data
      - Works well with agent decision audit trails

  # Qdrant scroll API for vector search pagination
  vector_pagination:
    description: "Scroll through large vector search results"
    pattern: |
      POST /collections/{collection}/points/scroll
      {
        "filter": { ... },
        "limit": 50,
        "offset": "${scroll_offset}",
        "with_payload": true,
        "with_vector": false
      }
    max_scroll_depth: 10000  # prevent deep pagination abuse
```
