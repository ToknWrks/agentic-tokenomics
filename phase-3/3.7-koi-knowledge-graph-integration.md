# Phase 3.7: KOI Knowledge Graph Integration

## Overview

KOI (Knowledge Organization Infrastructure) serves as the knowledge layer for the Regen Agentic Tokenomics system. It bridges the gap between raw on-chain state and the rich contextual understanding that autonomous agents require for informed decision-making. Where Ledger MCP provides real-time blockchain state, KOI provides the historical context, documentation, code intelligence, and semantic relationships that enable agents to reason about proposals, methodologies, credit classes, and ecosystem dynamics.

KOI is backed by two complementary storage engines:

- **Apache Jena** (RDF triple store) --- structured knowledge as an RDF graph with entities, relationships, provenance metadata, and ontology definitions. SPARQL endpoint for precise, composable queries.
- **Qdrant** (vector database) --- BGE-large 1024-dimensional embeddings for semantic similarity search across unstructured content (forum posts, documentation, code comments, agent decision narratives).

Together these power a hybrid retrieval system: exact graph traversals via SPARQL, approximate semantic search via vectors, or both for high-recall, high-precision retrieval.

### Current Scale

| Metric | Count | Notes |
|--------|-------|-------|
| Documents indexed | ~64,000 | Forum posts, docs, proposals, READMEs |
| Code entities indexed | ~78,000 | Functions, types, modules, interfaces |
| RDF triples | ~2.1M | Entity-relationship graph |
| Vector embeddings | ~142,000 | Documents + code entities |
| Named graphs | 12 | Partitioned by source and domain |

### MCP Tool Surface

| Tool | Purpose | Backend |
|------|---------|---------|
| `search` | Semantic search across all content | Qdrant + Jena |
| `resolve_entity` | Find canonical entity by label | Jena SPARQL |
| `sparql_query` | Execute arbitrary SPARQL queries | Jena SPARQL |
| `query_code_graph` | Navigate code entity relationships | Jena code graph |
| `get_entity_documents` | Retrieve documents linked to entity | Jena + Qdrant |

## Data Flow Architecture

Three data sources feed KOI, each with distinct pipelines and refresh cadences.

```
┌────────────────────────────────────────────────────────────────────┐
│                      DATA SOURCES                                   │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐          │
│  │  ON-CHAIN      │  │  AGENT        │  │  EXTERNAL     │          │
│  │  Regen Ledger  │  │  ElizaOS      │  │  Discourse,   │          │
│  │  (events)      │  │  (decisions)  │  │  GitHub       │          │
│  └───────┬───────┘  └───────┬───────┘  └───────┬───────┘          │
│          ▼                   ▼                   ▼                  │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐          │
│  │  Blockchain   │  │  KOI Object   │  │  Source       │          │
│  │  Indexer      │  │  Writer       │  │  Crawlers     │          │
│  └───────┬───────┘  └───────┬───────┘  └───────┬───────┘          │
└──────────┼──────────────────┼──────────────────┼───────────────────┘
           └──────────────────┼──────────────────┘
                              ▼
           ┌──────────────────────────────────────┐
           │  RDF Transform → Jena (TDB2)         │
           │  Embedding     → Qdrant (1024-dim)   │
           └──────────────────────────────────────┘
```

### Source 1: On-Chain (Blockchain Indexer)

The indexer subscribes to Regen Ledger events via WebSocket, transforming chain state to RDF.

```yaml
indexer:
  chain_id: regen-1
  rpc_endpoint: https://rpc.regen.network:443
  ws_endpoint: wss://rpc.regen.network:443/websocket
  event_subscriptions:
    - regen.ecocredit.v1.EventCreateClass   → koig:onchain
    - regen.ecocredit.v1.EventCreateProject → koig:onchain
    - regen.ecocredit.v1.EventCreateBatch   → koig:onchain
    - regen.ecocredit.v1.EventRetire        → koig:onchain
    - cosmos.gov.v1beta1.EventSubmitProposal → koig:governance
    - cosmos.gov.v1beta1.EventVote          → koig:governance
    - cosmos.staking.v1beta1.*              → koig:onchain
    - regen.data.v1.EventAttest             → koig:attestations
  full_sync: daily (2 AM UTC)
  validator_set_poll: 5min
```

### Source 2: Agent Outputs (KOI Object Writer)

Decisions and artifacts are written to KOI immediately after OODA workflow completion via an internal callback.

```yaml
agent_writer:
  endpoint: http://koi-writer.internal:3003/objects
  object_types:
    - regen.agent.decision         → koig:agent-decisions   (real-time)
    - regen.agent.reputation_signal → koig:agent-reputation  (real-time)
    - regen.agent.governance_analysis → koig:governance      (real-time)
    - regen.agent.market_report    → koig:market-data        (hourly batch)
```

### Source 3: External (Discourse, GitHub)

```yaml
crawlers:

    base_url: https://forum.regen.network
```

## Agent Query Patterns

Actual SPARQL queries used by each agent during OODA observation and orientation phases.

### AGENT-001: Registry Reviewer

**Methodology Lookup** (WF-RR-01, OBSERVE phase):

```sparql
PREFIX regen: <https://schema.regen.network/ontology#>
PREFIX rdfs:  <http://www.w3.org/2000/01/rdf-schema#>
PREFIX dcterms: <http://purl.org/dc/terms/>

SELECT ?class ?classLabel ?project ?projectLabel ?batch ?attestation ?attestationIri
WHERE {
  ?methodology rdfs:label ?methodLabel .
  FILTER(CONTAINS(LCASE(?methodLabel), LCASE("VM0042")))
  ?class regen:usesMethodology ?methodology ; rdfs:label ?classLabel .
  OPTIONAL {
    ?project regen:belongsToClass ?class ; rdfs:label ?projectLabel .
    OPTIONAL { ?batch regen:issuedUnder ?project ; regen:batchDenom ?batchDenom }
    OPTIONAL { ?attestation regen:attestsTo ?project ; regen:attestationIri ?attestationIri }
  }
}
ORDER BY ?classLabel LIMIT 200
```

**Duplicate Detection** (WF-RR-01, ORIENT phase):

```sparql
PREFIX regen: <https://schema.regen.network/ontology#>
PREFIX rdfs:  <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?existingClass ?classId ?classLabel ?methodLabel ?priorOutcome ?priorConfidence
WHERE {
  ?existingClass a regen:CreditClass ;
      regen:classId ?classId ; rdfs:label ?classLabel ;
      regen:hasCreditType ?ct .
  ?ct rdfs:label "Carbon" .
  OPTIONAL {
    ?existingClass regen:usesMethodology ?m . ?m rdfs:label ?methodLabel .
  }
  OPTIONAL {
    ?d a regen:AgentDecision ; regen:referencesEntity ?existingClass ;
       regen:decisionOutcome ?priorOutcome ; regen:confidenceScore ?priorConfidence .
  }
}
ORDER BY ?classId
```

### AGENT-002: Governance Analyst

**Proposal History** (WF-GA-01, OBSERVE phase):

```sparql
PREFIX regen:  <https://schema.regen.network/ontology#>
PREFIX regenv: <https://schema.regen.network/vocab/>
PREFIX rdfs:   <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?proposal ?proposalId ?label ?type ?status
       ?yesVotes ?noVotes ?abstainVotes ?vetoVotes ?submitTime
WHERE {
  ?proposal a regen:Proposal ;
      regen:proposalId ?proposalId ; rdfs:label ?label ;
      regenv:proposalType ?type ; regenv:proposalStatus ?status .
  FILTER(?type = "ParameterChange")
  OPTIONAL { ?proposal regenv:yesVotes ?yesVotes }
  OPTIONAL { ?proposal regenv:noVotes ?noVotes }
  OPTIONAL { ?proposal regenv:abstainVotes ?abstainVotes }
  OPTIONAL { ?proposal regenv:vetoVotes ?vetoVotes }
  OPTIONAL { ?proposal regenv:submitTime ?submitTime }
}
ORDER BY DESC(?proposalId) LIMIT 50
```

**Related Forum Discussions** (WF-GA-01, ORIENT phase):

```sparql
PREFIX regen:   <https://schema.regen.network/ontology#>
PREFIX regenr:  <https://schema.regen.network/resource/>
PREFIX rdfs:    <http://www.w3.org/2000/01/rdf-schema#>
PREFIX dcterms: <http://purl.org/dc/terms/>

SELECT ?post ?postLabel ?sourceUrl ?created
WHERE {
  ?post a regen:ForumPost ; rdfs:label ?postLabel ; regen:sourceUrl ?sourceUrl .
  {
    ?post regen:relatedProposal regenr:proposal-62 .
  } UNION {
    ?post dcterms:subject "governance" .
    FILTER(CONTAINS(LCASE(?postLabel), "proposal 62") || CONTAINS(LCASE(?postLabel), "#62"))
  }
  OPTIONAL { ?post dcterms:created ?created }
}
ORDER BY DESC(?created) LIMIT 30
```

**Credit Quality Context** (WF-MM-01, ORIENT phase):

```sparql
PREFIX regen:  <https://schema.regen.network/ontology#>
PREFIX rdfs:   <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?class ?classId ?classLabel ?attestationCount ?avgRepScore
WHERE {
  ?class a regen:CreditClass ; regen:classId ?classId ; rdfs:label ?classLabel .
  { SELECT ?class (COUNT(DISTINCT ?att) AS ?attestationCount) WHERE {
      ?proj regen:belongsToClass ?class . ?att regen:attestsTo ?proj .
  } GROUP BY ?class }
  OPTIONAL {
    SELECT ?class (AVG(?score) AS ?avgRepScore) WHERE {
      ?class regen:hasIssuer ?issuer .
      ?sig a regen:ReputationSignal ; regen:referencesEntity ?issuer ; regen:signalScore ?score .
    } GROUP BY ?class
  }
}
ORDER BY ?classId
```

## MCP Tool Integration

### Tool Signatures

```typescript
// search — hybrid semantic search
search(params: {
  query: string;
  intent?: "general" | "person_activity" | "technical_howto";
  limit?: number;
  source?: string;
}): Promise<{ results: Array<{ uri, title, content, score, source, metadata }>, total, query_time_ms }>

// resolve_entity — canonical entity lookup
resolve_entity(params: {
  label: string;
  type_hint?: string;
}): Promise<{ uri, label, type, confidence, alternatives[] }>

// sparql_query — arbitrary SPARQL SELECT (read-only, sandboxed)
```

### Error Handling

```typescript
export async function withKOIErrorHandling<T>(
  operation: () => Promise<T>,
  context: { tool: string; workflow: string; phase: string },
  options: { maxRetries?: number; backoffMs?: number } = {}
): Promise<T | null> {
  const maxRetries = options.maxRetries ?? 3;
  const backoffMs = options.backoffMs ?? 1000;
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await operation();
    } catch (error) {
      const koiError = parseKOIError(error);
      if (!koiError.retryable || attempt === maxRetries) {
        await recordKOIFailure(context, koiError);  // Audit trail
        return null;
      }
      const delay = koiError.retry_after_ms ?? backoffMs * Math.pow(2, attempt);
      await new Promise(resolve => setTimeout(resolve, delay));
    }
  }
  return null;
}
```

### Provenance Verification

The derivation chain enables backward tracing from decision to source data:

1. **Signature verification** --- secp256k1 over the decision hash
2. **Derivation integrity** --- each entity's integrity_hash matches KOI content
3. **On-chain verification** --- on-chain entities verified against ledger state at recorded block height
4. **Temporal ordering** --- UUID v7 encodes creation time for causal ordering

```
Decision → (signed by agent) → Derivation Chain → Source Entities → On-Chain State
    ↓                                ↓                                    ↓
  Queryable via SPARQL        Integrity hashes           Ledger block height
  in koig:agent-decisions     verify content             verifies state
```

### Hybrid SPARQL + Vector Search

The hybrid search pipeline combines structured graph queries with semantic vector search:

```typescript
async search(params: HybridSearchParams): Promise<HybridSearchResult[]> {
  // 1. Generate query embedding (BGE-large with instruction prefix)
  const queryVector = await this.embedder.embed(
    `Represent this sentence for searching relevant passages: ${params.query}`
  );

  // 2. Vector search in Qdrant (semantic similarity)
  const vectorResults = await this.qdrant.search(collection, {
    vector: queryVector,
    limit: params.limit * 2,
    score_threshold: 0.5,
    params: { hnsw_ef: 128 },
    filter: params.source ? { must: [{ key: "source", match: { value: params.source } }] } : undefined,
  });

  // 3. SPARQL search (structural relationships via label match)
  const sparqlResults = await this.jena.query(`
    SELECT ?entity ?label ?type WHERE {
      ?entity rdfs:label ?label ; a ?type .
      FILTER(CONTAINS(LCASE(?label), LCASE("${escapeSparql(params.query)}")))
    } LIMIT ${params.limit}
  `);

  // 4. Merge: vector results weighted 0.6, SPARQL 0.4
  //    Items appearing in both get boosted (hybrid match)
  const merged = mergeResults(vectorResults, sparqlResults, {
    vector_weight: 0.6, sparql_weight: 0.4
  });

  // 5. Optional cross-encoder re-rank (top 50 candidates)
  return rerank(merged).filter(r => r.score >= 0.65).slice(0, params.limit);
}
```

**Intent-specific weight overrides**:
- `technical_howto`: vector 0.7, SPARQL 0.3 (semantic similarity more valuable)
- `person_activity`: vector 0.4, SPARQL 0.6 (structural graph more valuable)
