# Phase 4.2: Mainnet Migration Plan

## Overview

This document specifies the complete migration plan for deploying the Regen Economic Reboot mechanisms (M008-M015) to Regen Ledger mainnet. It is the operational counterpart to the mechanism specifications in [Phase 2.6](../phase-2/2.6-economic-reboot-mechanisms.md) and the implementation specifications in [Phase 3.1](../phase-3/3.1-smart-contract-specs.md).

### Migration Philosophy

The migration follows four foundational principles:

1. **Staged deployment.** Mechanisms are activated in dependency order across four discrete stages. Each stage is validated on mainnet with live traffic before the next begins. The sequence mirrors the Phase 2.6 Implementation Sequence (M013 first, then M014+M012, then M015), prepended by CosmWasm contract deployment as a prerequisite stage.

2. **Safety-first.** Every stage includes explicit entry criteria, success criteria, and monitoring gates. No stage advances without verification that the prior stage is healthy. Testnet rehearsal precedes every mainnet action. AGENT-003 (Market Monitor) and AGENT-004 (Validator Monitor) run continuous monitoring workflows from block one of each activation.

3. **Reversible steps.** Each mechanism includes a pre-tested rollback procedure that can restore the prior network state. CosmWasm contracts can be migrated to no-op implementations or paused via admin key. Native module upgrades include halt-and-revert binaries. Parameter changes can be reversed via governance proposal. The system is designed so that disabling a later mechanism never breaks an earlier one: the dependency graph is acyclic.

4. **Governance-gated activation.** Every deployment step that modifies chain behavior requires an on-chain governance proposal. No mechanism activates without explicit community consent via the governance process formalized in [Phase 2.3](../phase-2/2.3-governance-processes.md). Constitutional changes (M012 hard cap, M014 validator set structure) require Layer 4 supermajority (67%).

### Guiding Constraints

| Constraint | Rationale |
|-----------|-----------|
| No mechanism activates without passing governance | Constitutional requirement; protocol-level changes require on-chain approval |
| CosmWasm contracts deploy before native module upgrades | Lower risk; contracts are sandboxed and independently upgradeable |
| Fee routing (M013) activates before PoA (M014) or Dynamic Supply (M012) | M013 provides the revenue stream that funds both validator compensation and burn input |
| PoA transition spans 6-12 months minimum | Allows delegators adequate time to unbond and prevents economic shock |
| Every activation includes AGENT-004 monitoring from block 1 | Automated anomaly detection provides the earliest possible signal of issues |

## Governance Proposal Templates

All proposals follow Regen Ledger's `x/gov` proposal format. Deposit: 200 REGEN (current mainnet parameter). Voting period: 7 days (604,800 seconds).

### Upgrade Height Calculation

```yaml
upgrade_height_calculation:
  method: >
    target_date is the desired activation date.
    current_height is the latest block at time of proposal submission.
    block_time is the average block time (currently ~6.5 seconds on Regen mainnet).

    upgrade_height = current_height + ((target_date - now) / block_time)

    Round UP to nearest 1000 for clean upgrade points.
    Add a 48-hour buffer beyond the voting period end to allow
    validator preparation.

  example: >
    If current_height = 18,500,000 and target is 14 days from now:
    blocks_in_14_days = (14 * 86400) / 6.5 = 185,846
    upgrade_height = 18,500,000 + 185,846 = 18,685,846 -> round to 18,686,000
```

### a) PROP-M013-UPGRADE: M013 Software Upgrade Proposal

```
Title: Enable Value-Based Fee Routing (M013)

Description:

## Summary

This proposal upgrades Regen Ledger to v6.0, introducing the x/feerouter
module that implements M013 Value-Based Fee Routing as specified in the
Economic Reboot Roadmap.

## What Changes

- New module: x/feerouter intercepts credit transactions (MsgCreateBatch,
  MsgSend, MsgRetire, MsgBuySellOrder) and applies value-proportional fees.
- New module accounts: BurnPool, ValidatorFund, AgentInfraFund are created
  (CommunityPool already exists).
- Backward compatible: Non-credit transactions retain flat gas fees. All
  existing wallet software, block explorers, and API integrations continue
  to function without modification.
- Dual-fee mode: For the first 90 days, both flat gas AND value-based fees
  apply to credit transactions. Value-based fees are collected and routed
  to purpose-specific pools, while flat gas fees continue to compensate
  validators under the current PoS model. After 90 days, a follow-up
  governance proposal (PROP-M013-CONFIG) transitions to value-based-only
  mode for credit transactions.

## Why This Upgrade

Currently, a $10 credit transaction and a $10,000 credit transaction both
pay the same ~0.01 REGEN gas fee. This disconnect between protocol revenue
and facilitated value makes the network economically unsustainable without
perpetual inflation. M013 creates a sustainable revenue model where protocol
income scales with the ecological value flowing through the network.

This is the foundational mechanism for the Economic Reboot Roadmap. It
provides the revenue streams that fund:
- Token supply management via M012 burn pool
- Authority validator compensation via M014 validator fund
- Contributor rewards via M015 community pool distribution
- Agent infrastructure operations

## Impact on Users

- Credit issuers: 1-3% fee on credit value at issuance (configurable)
- Credit buyers/sellers: 1% marketplace fee on trade value
- Credit transfers: 0.1% fee on transfer value
- Credit retirers: 0.5% fee on retirement value
- All other transactions: NO CHANGE (standard flat gas)

Fee rates are NOT activated by this upgrade. A separate parameter change
proposal (PROP-M013-CONFIG) will configure and activate fee rates after
the module is confirmed operational during the 90-day dual-fee period.

## Security

- Audited by [AUDITOR] (report: [LINK])
- All security invariants from Phase 3.4 verified:
  Fee Conservation, Share Sum Unity, Non-Negative Fees,
  Rate Bound Safety (max 10% per rate), Pool Isolation
- Testnet validation: [TESTNET_BLOCK_RANGE] with [N] credit transactions
  processed without error

## Upgrade Details

- Binary: regen-ledger v6.0.0
- Upgrade name: "v6.0-fee-routing"
- Upgrade height: [UPGRADE_HEIGHT] (calculated per formula above)
- Module additions: x/feerouter
- State migrations: FeeConfig, PoolBalance initialized with zero values
- Breaking changes: None (fee rates default to 0 until configured)

## Rollback

If critical issues are discovered post-upgrade, the validator set can
coordinate a halt-and-rollback to the pre-upgrade binary. The x/feerouter
module state is safely ignored by the prior binary version.

## References

- Specification: phase-2/2.6-economic-reboot-mechanisms.md, M013 section
- Smart contract specs: phase-3/3.1-smart-contract-specs.md

### c) PROP-M014-M012-UPGRADE: Combined PoA + Dynamic Supply Upgrade

```
Title: Enable Authority Validator Governance (M014) and Fixed Cap Dynamic
       Supply (M012)

Description:

## Summary

This proposal upgrades Regen Ledger to v7.0, introducing the x/authority
module (M014 Authority Validator Governance) and the x/supply module
(M012 Fixed Cap Dynamic Supply). These two mechanisms are deployed together
because they are tightly coupled: M012 requires M014 to be active so that
inflation can be safely disabled, and M014 requires M012 so that the
staking_multiplier can transition to the stability_multiplier as PoS is
sunset.

## What Changes

### M014 -- Authority Validator Governance

- New module: x/authority for curated validator set management
- Introduces validator application, approval, term lifecycle, and
  compensation model
- Validator compensation sourced from M013 Validator Fund (replacing
  inflationary staking rewards over the transition period)
- Three validator composition categories: Infrastructure Builders (min 5),
  Trusted ReFi Partners (min 5), Ecological Data Stewards (min 5)
- Target set size: 15-21 authority validators
- AGENT-004 (Validator Monitor) integration for performance tracking
- Validator term length: 12 months with quarterly compensation review

### M012 -- Fixed Cap Dynamic Supply

- New module: x/supply implementing algorithmic mint/burn
- Hard cap set at 221,000,000 REGEN (constitutional parameter, Layer 4)
- Supply algorithm: S[t+1] = S[t] + M[t] - B[t]
- Minting (regrowth): M[t] = r_base * (C - S[t]) * effective_multiplier
  * ecological_multiplier
- r_base = 0.02 (2% base regrowth rate per period)
- ecological_multiplier disabled in v0 (set to 1.0 until oracle available)
- Burning: B[t] = tokens from M013 Burn Pool permanently destroyed per epoch
- Disables x/mint inflation module (inflation parameters set to read-only)

## Why Combined Upgrade

Deploying M014 and M012 together avoids an intermediate state where one
mechanism is active without the other. Specifically:
- M012 cannot safely disable inflation until M014 provides an alternative
  compensation model for validators (fee-based rather than inflation-based)
- M012's effective_multiplier uses a max(staking, stability) coexistence
  function during the PoA transition (Phase 2.6, M012 section), which
  requires M014 state to determine which multiplier mode is active

## Prerequisite Verification

- Full PoA transition rehearsal completed on testnet: [DATE]
- Security audit completed: [DATE], [AUDITOR], no critical findings

## Impact on Users

  transition follows a 3-phase runbook (see Phase 4.2 PoS-to-PoA Transition
  Runbook). During Phase 1 (Parallel Operation, 3-6 months), existing
  stakers continue to earn inflation rewards while authority validators
  are onboarded.
- Token supply: The hard cap is set but active mint/burn begins only after
  M014 reaches TRANSITION state and the first burn period completes.
- Governance: Voting continues under current PoS model during Phase 1.
  Dual-track tally (60% validator / 40% holder) activates via separate
  governance proposal during Phase 2.

## Upgrade Details

- Binary: regen-ledger v7.0.0
- Upgrade name: "v7.0-poa-supply"
- Upgrade height: [UPGRADE_HEIGHT]
- Module additions: x/authority, x/supply
- Module modifications: x/staking (parameter restrictions), x/mint (read-only)

  - AuthorityValidator set initialized from approved applications
  - SupplyState initialized: circulating_supply = current total,

## Rollback

## PoS to PoA Transition Runbook

The transition from Proof of Stake to Proof of Authority is the most operationally complex step in the migration. It spans 6-12 months and is divided into three phases, each gated by explicit entry criteria, actions, and success criteria. This runbook is executed within Stage 3 after the M014+M012 upgrade proposal passes.

### Transition State Machine

│   OPERATION      │     │   TALLY          │     │                  │

## Rollback Procedures

Every mechanism activation includes a pre-tested rollback path. Rollbacks are executed via emergency governance proposals (Layer 4 fast-track where supported) or, in extreme cases, via coordinated validator action.

### General Rollback Principles

```yaml
rollback_principles:
  1_no_data_loss: >
    Rollbacks disable mechanism logic but preserve all historical data
    (fee records, supply snapshots, activity scores). Data can be used
    to debug issues and inform re-activation.
  2_independent_rollback: >
    Each mechanism can be rolled back independently without affecting
    earlier mechanisms, following the dependency ordering:
    M015 -> M012 -> M014 -> M013 -> M008-M011.
    Rolling back a dependency mechanism automatically pauses dependents.
  3_governance_authority: >
    Rollback proposals require Layer 4 governance (67% supermajority)
    unless a security emergency is declared by the authority validator set.
  4_communication_first: >
    No rollback executes without a public announcement at least 24 hours
    prior (except for critical security vulnerabilities requiring
    immediate action).
```

### Critical Path

```
CosmWasm Deploy ──► M013 Upgrade ──► M013 Config ──► M014+M012 Upgrade ──►
     (2mo)            (3.5mo)          (0.5mo)           (0.5mo)
                                                            │
                                                            ▼
    M015 Distrib. ◄── M015 Track ◄── M015 Upgrade ◄── PoA Complete
       (ongoing)        (3mo)           (0.5mo)        (7-13mo)
```

The critical path is dominated by the PoA transition (Stage 3), which requires 7-13 months of staged operation. This is by design: the PoS-to-PoA transition is the highest-risk change in the migration and requires the longest observation period to ensure network stability, validator readiness, and delegator safety.

Strategies to compress the total timeline without compromising safety:

1. **Parallelize audit work**: Begin M014/M012 audit during the Stage 2 dual-fee observation period
2. **Overlap communication**: Begin authority validator outreach and applications during Stage 2
3. **Compress PoA Phase 1**: If validator applications and approval proceed quickly, Phase 1 can be shortened to 3 months (minimum)
4. **Begin M015 audit during Stage 3**: M015 audit and testnet preparation can proceed during the PoA transition

The irreducible minimum timeline is approximately 14 months, assuming all stages proceed at minimum durations, audits are parallelized, and no rollbacks are needed.
